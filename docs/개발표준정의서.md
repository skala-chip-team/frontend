# 개발표준정의서 (UI)

본 문서는 프론트엔드 UI 영역의 구조와 개발 표준을 정의한다. 대상 시스템은 스마트 팩토리 생산 관제 화면이며, 기술 스택은 Vite, React 19, TypeScript, Tailwind CSS, Zustand, TanStack Query(React Query), Axios, React Router 로 구성된다. 원래 표준 템플릿의 WebSquare 와 Rexpert 항목은 본 프로젝트가 사용하는 실제 도구에 맞춰 제목과 내용을 대체한다.

---

## 3.8 UI 구조

UI 구조는 소스 폴더 배치, 애플리케이션 실행 구성, 화면 구성, 데이터 수집 계층의 네 가지로 나누어 정의한다.

### 3.8.1 프로젝트 구조

소스는 역할별 폴더로 나누고, 각 폴더는 `index.ts` 배럴 파일로 외부 노출 범위를 관리한다. 공통으로 재사용하는 UI 요소는 `components/common` 에 모은다.

```
src/
  apis/          서버 통신 모듈과 응답 타입(DTO) 정의
  assets/        이미지, 아이콘, 로고 등 정적 자원
    icons/
    logo/
  components/
    common/      공용 UI 컴포넌트 (테이블, 모달, 칩, 차트 등)
    three/       3D 시각화 컴포넌트 (공장, 장비 플릿)
    reschedule/  재조정 화면 전용 컴포넌트
  hooks/         React Query 훅과 화면 보조 훅
  mocks/         API 미연결 시 사용하는 목업 데이터
  pages/         라우트 단위 페이지 컴포넌트
  routes/        라우터 정의와 접근 가드
  stores/        Zustand 클라이언트 상태 스토어
  types/         도메인 타입 정의
  utils/         변환 함수와 공용 유틸
  reactQuery.ts  QueryClient 인스턴스
  main.tsx       앱 진입점
  index.css      Tailwind 설정과 디자인 토큰
```

폴더와 파일 명명 규칙은 아래 표를 따른다.

| 구분 | 규칙 | 예시 |
| --- | --- | --- |
| 디렉터리 | 소문자 하이픈 | user-profile/ |
| 컴포넌트 파일 | PascalCase.tsx | MachineTable.tsx |
| 페이지 파일 | PascalCase.tsx | DashboardPage.tsx |
| 훅 파일 | camelCase.ts | useMachines.ts |
| 일반 파일 | camelCase.ts | dashboardTransform.ts |
| 변수, 함수 | camelCase | handleSubmit |
| 컴포넌트, 타입 | PascalCase | MachineConfig |

경로 별칭(alias)을 두어 상대 경로 깊이를 줄인다. Vite 와 TypeScript 양쪽에 동일하게 설정한다.

| 별칭 | 실제 경로 |
| --- | --- |
| @ | src |
| @apis | src/apis |
| @components | src/components |
| @hooks | src/hooks |
| @pages | src/pages |
| @routes | src/routes |
| @stores | src/stores |
| @utils | src/utils |
| @assets | src/assets |
| @styles | src/styles |

배럴 파일 사용 원칙은 다음과 같다. 폴더 외부에서 쓰는 항목만 `index.ts` 에서 내보내고, 내부 전용 구현은 내보내지 않는다. import 는 가급적 별칭과 배럴을 통해 한다.

### 3.8.2 애플리케이션 구성 (Vite, React)

빌드와 개발 서버는 Vite 를 사용한다. 개발 서버는 핫 모듈 교체(HMR)를 제공하고, 프로덕션 빌드는 정적 번들을 생성한다.

플러그인 구성은 다음과 같다.

| 플러그인 | 역할 |
| --- | --- |
| @vitejs/plugin-react | React 와 JSX, Fast Refresh |
| @tailwindcss/vite | Tailwind CSS v4 통합 |
| vite-plugin-svgr | SVG 를 React 컴포넌트로 import |

앱 진입점은 `main.tsx` 이며 전역 Provider 계층은 아래 순서로 감싼다. 최상단에 React Query 의 QueryClientProvider 를 두고, 그 안에 React Router 의 RouterProvider 를 둔다.

```
StrictMode
  QueryClientProvider (queryClient)
    RouterProvider (router)
```

라우팅은 React Router 의 createBrowserRouter 로 정의한다. 인증과 권한은 가드 컴포넌트로 분리한다.

| 경로 | 화면 | 가드 |
| --- | --- | --- |
| /login | 로그인 | 없음 |
| /signup | 회원가입 | 없음 |
| /dashboard | 대시보드 | RequireAuth |
| /orders | 주문 관리 | RequireAuth |
| /reschedule | 재조정 제안 관리 | RequireAuth |
| /workers | 작업자 관리 | RequireAuth, RequireAdmin |
| /machines | 장비 설정 | RequireAuth, RequireAdmin |

로그인 이후 화면은 공통 레이아웃(SidebarLayout) 아래에 중첩 라우트로 배치한다. SidebarLayout 은 좌측 사이드바와 상단 헤더, 본문 아웃렛으로 구성된다.

개발 환경에서 백엔드 호출은 Vite 프록시로 우회한다. 운영 서버는 환경 변수로 분리한다.

| 프록시 경로 | 대상 |
| --- | --- |
| /api | 백엔드 모니터링 서버(기본 localhost:8080) |
| /sim | 시뮬레이션 서버(기본 localhost:8000) |

### 3.8.3 화면 구조

화면은 인증 화면과 업무 화면으로 나뉜다. 업무 화면은 공통 레이아웃(사이드바, 헤더, 본문) 위에 페이지를 끼우는 형태다. 본 절에서는 재조정 제안 목록과 상세 화면은 정의 범위에서 제외한다.

공통 레이아웃 구조는 다음과 같다.

```
┌───────────┬──────────────────────────────────────────┐
│           │  헤더 (구역 선택, 예측 상태, 알림)         │
│ 사이드바   ├──────────────────────────────────────────┤
│ (메뉴)     │                                          │
│           │            본문 (라우트 페이지)            │
│           │                                          │
└───────────┴──────────────────────────────────────────┘
```

[공통 레이아웃 화면]

화면 목록과 계층은 아래와 같다.

1. 인증
   - 로그인 [로그인 화면]
   - 회원가입 [회원가입 화면]

2. 공장 생산 관제
   - 전체 대시보드. 전체 구역을 3D 로 보여주고, 구역을 클릭하면 줌인과 함께 구역 요약, 장비 상세, 대기 큐를 표시한다. [전체 대시보드 화면]
   - 구역별 대시보드. 생산 달성률, 요약 카드, 공정 STEP 선택, 3D 장비 플릿, 하단 스케줄 간트(계획, 현재 상태 탭)로 구성한다. [구역별 대시보드 화면]
   - 장비 상세. 3D 에서 장비를 클릭하면 우측에서 슬라이드되어 가동률, 부하율, 진행 중 유닛을 보여준다. [장비 상세 화면]
   - 주문 관리 목록. 일일 주문을 표로 조회하고 상태 필터, 정렬, 페이지네이션을 제공한다. [주문 목록 화면]
   - 주문 상세. 선택한 주문의 정보와 유닛별 공정 진행을 우측 패널로 보여준다. [주문 상세 화면]

3. 운영 관리
   - 작업자 관리. 작업자 목록을 역할 필터로 조회한다. [작업자 관리 화면]
   - 작업자 상세. 역할과 권한 구역을 우측 패널에서 수정한다. [작업자 상세 화면]
   - 장비 설정. 장비 목록을 구역과 STEP 으로 필터하고 추가, 삭제, 공정 매핑을 한다. [장비 설정 화면]
   - 장비 추가. 타입, 구역, 담당 STEP 을 입력하는 모달이다. [장비 추가 화면]

화면 공통 상태 표기 원칙은 다음과 같다. 로딩 중에는 안내 박스 또는 스켈레톤을 보여주고, 오류 시에는 오류 안내를 보여주며, 데이터가 없으면 빈 상태 문구를 보여준다. 무거운 3D 화면은 진입 직후 가벼운 스켈레톤을 먼저 그리고, 첫 페인트 이후에 마운트하여 메뉴 반응을 우선한다.

### 3.8.4 데이터 수집(DataCollection) 구조

데이터 수집 계층은 서버 상태와 클라이언트 상태를 분리하여 관리한다. 서버 상태는 React Query 로, 화면 전용 클라이언트 상태는 Zustand 로 다룬다. 서버 데이터를 Zustand 에 넣지 않는다.

데이터 흐름은 다음 단계로 구성한다.

```
1) apis/        Axios 인스턴스로 엔드포인트 호출, 응답 DTO 수신
2) utils/       DTO 를 화면용 도메인 타입으로 변환(transform)
3) hooks/       React Query 로 캐시, 로딩, 오류, 폴링 관리
4) pages/       훅이 돌려준 데이터를 컴포넌트에 주입
```

각 계층의 표준은 다음과 같다.

1. 통신 모듈(apis)
   - 공용 Axios 인스턴스(apiClient)를 사용한다. 기본 헤더와 토큰 첨부, 타임아웃을 인스턴스에 둔다.
   - 인증 토큰은 요청 인터셉터에서 Bearer 로 첨부한다.
   - 응답은 공통 래퍼(success, code, message, data) 형태를 가정하고 data 만 꺼내 반환한다.

2. 변환(utils)
   - 백엔드 DTO 는 보통 camelCase 이고, 화면 도메인 타입은 docs/data.dbml 컬럼명을 따라 snake_case 로 둔다. 변환 함수에서 둘 사이를 매핑한다.
   - 시각 데이터는 표시에 맞게 가공한다. 예로 ISO 시각을 시(소수)로 바꾸거나 날짜 기준으로 거른다.

3. 서버 상태(hooks, React Query)
   - 조회는 useQuery, 변경은 useMutation 으로 작성하고, 변경 성공 시 관련 쿼리를 무효화한다.
   - QueryClient 기본값은 staleTime 5분, retry 1회, 창 포커스 시 자동 재조회 비활성으로 둔다.
   - 실시간성이 필요한 화면은 폴링 주기를 둔다. 시뮬레이션이 도는 동안에만 짧은 주기로 갱신하고 멈추면 폴링을 중단한다.
   - 백엔드가 아직 없는 기능은 목업(mocks)으로 동작시키고, 같은 훅 구조를 유지하여 실제 API 연결 시 호출부만 교체한다.

4. 클라이언트 상태(stores, Zustand)
   - 선택한 구역, 모달 열림 여부, 알림 토스트 같은 화면 상태만 둔다.
   - 서버에서 받은 목록이나 상세 데이터는 두지 않는다.

## 3.9 UI 개발 표준

UI 개발 표준은 화면 컴포넌트 작성 표준과 데이터 시각화 표준으로 나눈다.

### 3.9.1 React 와 Tailwind 컴포넌트 표준

원래 템플릿의 WebSquare 항목에 해당하며, 본 프로젝트의 화면 구현 기술인 React 와 Tailwind 기준으로 정의한다.

1. 컴포넌트 작성
   - 함수형 컴포넌트로 작성하고 props 는 TypeScript 인터페이스로 명시한다.
   - 한 파일은 하나의 책임만 갖도록 작게 유지한다. 파일이 커지면 분리한다.
   - 외부에서 쓰는 컴포넌트와 타입만 배럴로 내보낸다.
   - React 17 이상 자동 import 를 사용하므로 별도 React import 는 하지 않는다.

2. 상태 관리 기준
   - 서버 상태는 React Query, 화면 클라이언트 상태는 Zustand 로 분리한다.
   - 폼 입력값처럼 일시적인 값은 지역 상태(useState)로 둔다.
   - 효과(useEffect) 안에서 상태를 즉시 설정하는 방식은 피하고, 렌더 중 파생 계산이나 키 기반 초기화로 대체한다.

3. 스타일과 디자인 토큰
   - 모든 스타일은 Tailwind 유틸리티로 작성하고, 색과 타이포는 index.css 의 디자인 토큰을 사용한다. 임의 색상값을 직접 쓰지 않는다.
   - 주요 토큰 군은 primary(브랜드 레드 계열), secondary-navy, secondary-orange, surface(배경 계열), gray 계열이다.
   - 타이포는 heading, subtitle, body, label 등 단계 토큰을 쓰고 폰트는 Pretendard 를 사용한다.
   - 컴포넌트는 사이드바가 정한 시각 언어(둥근 모서리, 옅은 그림자, 카드형 레이아웃)와 일치시킨다.

4. 아이콘
   - 아이콘은 lucide-react 단색 라인 아이콘만 사용하고 currentColor 를 상속한다.
   - 컬러 이모지나 색이 칠해진 아이콘은 사용하지 않는다.

5. 공통 상호작용 패턴
   - 추가와 수정 입력은 가운데 모달(Modal)을 사용한다.
   - 상세 보기와 편집은 우측 슬라이드 패널을 사용한다.
   - 삭제처럼 되돌릴 수 없는 동작은 확인 모달(ConfirmModal)을 거친다.
   - 목록은 공통 테이블 스타일(헤더 옅은 배경, 행 구분선, 행 진입 애니메이션)을 따른다.

코드 품질과 린트 규칙은 3.9.3 에서 별도로 정의한다.

### 3.9.2 데이터 시각화 표준

원래 템플릿의 Rexpert(리포트) 항목에 해당하며, 본 프로젝트는 별도 리포트 도구 대신 React 기반 커스텀 시각화와 3D 렌더링을 사용한다.

1. 3D 시각화
   - 3D 는 @react-three/fiber 와 three, 보조로 @react-three/drei 를 사용한다.
   - 전체 대시보드의 공장 뷰와 구역별 장비 플릿이 여기에 해당한다.
   - 카메라는 고정 각도에서 대상 수에 따라 줌 거리를 정하고, 선택 시 해당 대상으로 줌인한다.
   - 성능을 위해 무거운 캔버스는 화면 진입 직후가 아니라 첫 페인트 이후에 마운트한다. 마운트 전에는 옅은 스켈레톤을 보여준다.

2. 스케줄 간트
   - 간트는 커스텀 컴포넌트로 작성한다. 가로축은 0시에서 24시, 빨간 현재선은 시뮬레이션 시각에 맞춘다.
   - 계획과 현재 상태를 탭으로 전환한다. 계획은 계획 시작일 기준, 현재 상태는 실제 작업일 기준으로 표시한다.
   - 막대는 장비 행에 배치한다. 계획은 계획 장비, 현재 상태는 실제 투입 장비 기준으로 둔다.
   - 막대가 좁아 글씨가 가려지면 마우스 오버 시 풀네임과 시간을 툴팁으로 보여준다.

3. 커스텀 차트와 지표
   - 원형 진행률(CircularProgress)은 부하율 같은 단일 비율 지표에 사용한다.
   - 가로 진행 바(ProductionAchievementBar)는 생산 달성률에 사용한다.
   - 레이더(StrategyRadar)는 전략 비교 같은 다축 지표에 사용한다.
   - 차트 색은 디자인 토큰을 따르고 상태 의미를 일관되게 매핑한다. 예로 가동은 초록, 점검은 주황, 정지는 빨강, 대기는 회색이다.

4. 실시간 표현
   - 시뮬레이션이 도는 동안 데이터를 주기적으로 갱신하고, 진행 중 항목은 현재 시각까지만 그린다.
   - 갱신으로 화면이 흔들리지 않도록 표시 기준(날짜, 장비)을 고정하여 불필요한 추가나 제거가 일어나지 않게 한다.

5. 입력 보조
   - 날짜 입력은 react-datepicker 를 사용하고 디자인 토큰에 맞춰 스타일을 맞춘다.

### 3.9.3 코드 품질과 린트 표준

코드 품질 도구는 ESLint, Prettier, TypeScript 컴파일러를 사용한다. 작업 도중 자동으로 실행되며 통과해야 작업을 완료로 본다.

1. Prettier 포맷팅
   - printWidth 100, tabWidth 2, useTabs false
   - 세미콜론 사용, 작은따옴표(singleQuote), trailingComma es5, bracketSpacing true, arrowParens always
   - endOfLine auto

2. ESLint 규칙
   - react-hooks/rules-of-hooks 를 오류로 강제한다
   - react-hooks/exhaustive-deps 는 경고로 둔다
   - no-unused-vars 는 오류로 두되 언더스코어 접두 매개변수는 허용한다
   - react/react-in-jsx-scope 는 off (React 17 이상 자동 import)
   - import 해석은 alias resolver 를 사용한다
   - effect 안에서 직접 setState 하는 패턴은 경고로 잡고, 렌더 중 파생 계산이나 키 기반 초기화로 대체한다

3. 타입 검사
   - tsc --noEmit 로 타입 오류 0 을 유지한다
   - any 사용을 지양하고 도메인 타입을 명시한다. 서버 응답 DTO 와 화면 도메인 타입을 분리한다

4. 검사 시점과 완료 기준
   - tsc 와 eslint 는 작업 중 자동으로 실행되므로 별도 수동 실행은 필요하지 않다
   - 시각적 변경이 있으면 개발 서버를 띄워 실제 화면에서 확인한 뒤 완료로 본다
   - 통과 증거 없이 완료로 보고하지 않는다

5. 적용 예정(보류)
   - 커밋 전 자동 검사(pre-commit) 도입은 보류 항목으로 둔다
